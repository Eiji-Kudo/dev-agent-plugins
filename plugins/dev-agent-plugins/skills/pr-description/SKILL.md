---
name: pr-description
description: PR description生成・更新
---
現在のブランチに対応するPRの内容を取得し、差分を丁寧に分析した上でMarkdownファイルとして保存し、GitHubのPRを更新してください。

## 手順

1. **PR情報の取得**
   - `gh pr view --json title,body,number,headRefName,baseRefName` で現在のブランチに対応するPR情報を取得
   - PRが見つからない場合はエラーメッセージを表示して終了

2. **PRテンプレートの確認**
   - `.github/pull_request_template.md` が存在する場合は内容を確認
   - テンプレートの構造に従ってPR内容を整理

3. **変更ファイルの全体把握**
   - `gh pr diff --name-only` で変更ファイルのパスを取得
   - `gh pr diff` で全体の差分を取得
   - 変更をカテゴリ別に分類（バックエンド/フロントエンド/テスト/設定/ドキュメント等）

4. **既存アーキテクチャの理解**
   - CLAUDE.mdやREADME.mdがあれば確認し、プロジェクト構造を把握
   - 変更ファイルの周辺コード（同じディレクトリの既存ファイル等）を確認
   - 変更が既存のパターンや規約に沿っているか理解

5. **差分の詳細分析（重要）**
   - `gh pr diff` の出力からファイルごとの差分を確認
   - 各ファイルで「何が追加/変更/削除されたか」を正確に把握
   - 単なるファイル名の羅列ではなく、変更の意図・目的を理解
   - 以下の観点で分析:
     - 新規追加されたクラス/関数/コンポーネント
     - 既存コードへの修正内容
     - 削除された機能
     - 依存関係の変更
     - 設定値の変更

6. **変更内容の要約・整理（重要）**
   - 差分分析で得た情報を、レビュアーが理解しやすい形に整理
   - ファイル単位ではなく「機能単位」「目的単位」でグルーピング
   - 例: 「認証機能の追加」として、Controller/Service/Model/Testをまとめて説明
   - 細かい変更（import文の追加、型定義の更新等）は省略し、本質的な変更に焦点を当てる
   - PRの「Why（なぜ）」と「What（何を）」が明確に伝わるように要約
   - 以下の構造で整理:
     - このPRで実現したこと（1-2文の概要）
     - 主要な変更点（機能・目的ごとにグルーピング）
     - 技術的な詳細（必要な場合のみ）
   - **変更点の箇条書きは必ず5行以内に収めること**
     - 5行を超える場合は、実行環境のサブエージェント機能で要約専用agentを起動する
     - agentには差分分析の全結果を渡し、以下のプロンプトで要約させる:
       「以下のPR変更内容を、レビュアーが全体像を素早く把握できるよう5行以内の箇条書きに要約してください。個別ファイルの羅列ではなく、機能・目的の単位で抽象度を上げてまとめること。各行は1文で簡潔に。CIから自明な情報（CIが通っている、OpenAPI再生成、lint修正、自動生成ファイルの更新等）は含めないこと。」
     - agentの出力をそのまま変更点セクションとして採用する

7. **PR内容の作成**
   - 要約・整理した内容を基に、テンプレートに従ってMarkdown形式で作成
   - **タイトルは日本語で記述する**（例: `# [fix] EKS管理ノードグループ設定のlintフォーマット修正`）
   - フォーマルな文体で記述（敬語は使用しない）
   - 変更点は具体的かつ簡潔に（冗長な羅列は避ける）
   - 変更の背景・目的が分かるように記述
   - **フォローアップ issue セクションを必ず含める**: この PR の作業中に作成した follow-up issue（critics の「対応不要→別 issue」、型安全性/ドリフトの issue 化、レビュー由来の別 issue 等）を `## フォローアップ issue` セクションに列挙する。各 issue は `- #<番号> <タイトル>` の形式で、**`https://github.com/<owner>/<repo>/issues/<番号>` のフル URL リンク付き**にする（プレーンな `#<番号>` で終わらせない）。作成した issue が無ければ「なし」と明記し、セクション自体は省略しない

8. **関連issueの検索**
   - `gh issue list` や `gh search issues` を使用して、PRのタイトルや内容に関連しそうなissueを検索
   - 関連issueが見つかった場合は、issueの内容（`gh issue view`）とPRの差分を照合し、このPRでissueの要件がすべて満たされるか判断する
   - issueが完全に解決される場合のみ `close {issueのURL}` の形式でリンクを追加する
   - 部分的にしか対応していない場合や判断が難しい場合は、`close` を付けずにURLのみをリンクする

9. **動作確認チェックリストの作成**
   - 差分分析の結果から、マージ前に最低限確認すべき動作確認項目を洗い出す
   - 以下の観点で項目を抽出:
     - 変更した機能の正常系動作（新規追加・修正した主要フローが期待通り動くか）
     - 変更した機能の異常系・エッジケース（バリデーション、エラーハンドリング等）
     - 既存機能へのデグレがないか（変更の影響を受ける既存機能の確認）
     - UI変更がある場合はレイアウト崩れ・レスポンシブ対応
     - API変更がある場合はリクエスト/レスポンスの互換性
     - DB変更がある場合はマイグレーションの実行確認
     - 環境変数・設定値の変更がある場合は反映確認
   - 箇条書きのチェックリスト形式（`- [ ]`）で記述
   - 自動テストでカバーされている項目は除外し、手動でしか確認できない項目に絞る
   - **CIから自明な項目は書かない**（lint/format通過、型チェック通過、ユニットテスト通過、ビルド成功、自動生成ファイルの整合性、OpenAPI再生成結果等）
   - 確認項目は具体的な操作手順レベルで記述（例: 「ログイン画面でメールアドレス未入力のまま送信ボタンを押し、バリデーションエラーが表示されること」）

10. **ファイル保存**
   - 作成した内容をMarkdownファイルとして保存
   - ファイルの先頭にタイトル行を `# [種別] タイトル` の形式で追加する
   - ファイル名は `pr-{PRの番号}.md` とする（例: `pr-1730.md`）
   - **保存先はプロジェクトルートとする**

11. **GitHubのPR更新**
   - 保存したMarkdownファイルからタイトルを抽出（最初の `# ` で始まる行から `# ` プレフィックスを除去）
   - 本文はMarkdownファイルの内容からタイトル行を除いた部分を使用（先頭の空行も除去）
   - `gh pr edit {PR番号} --title "{タイトル}" --body "{本文}"` でPRを更新
   - bodyはHEREDOCで渡す（改行やクォートを正しく処理するため）
   - **macOS互換性**: テキスト加工には `tail -n +2 | awk 'NF{found=1} found'` を使用（BSD sedの互換性問題を回避）

12. **更新内容の検証・クリーンアップ**
   - `gh pr view --json title,body` でGitHub上のPRを再取得し、タイトルとbodyが正しく反映されていることを確認
   - 反映が確認できた場合のみ、ローカルの `pr-{PR番号}.md` ファイルを削除
   - 反映に問題がある場合はファイルを残し、ユーザーに報告する

13. **動作確認項目の実施**
   - ステップ9で作成した動作確認チェックリストの各項目について、CLIやコードリーディングで確認可能なものを実際に検証する
   - 確認可能な項目の例:
     - マイグレーションファイルの整合性チェック（カラム定義とモデルの一致等）
     - 設定ファイルの値が正しく反映されているか（ファイル読み取りで確認）
     - APIスキーマの整合性（OpenAPI定義と実装の一致等）
     - 環境変数の参照箇所が正しいか（コードリーディングで確認）
     - コマンドやスクリプトの実行による動作確認（ドライラン等）
   - 確認できた項目は `- [x]` に更新し、確認結果を簡潔に付記する（例: `- [x] マイグレーションとモデルのカラム定義が一致すること → コード比較で確認済み`）
   - CLI等では確認できない項目（ブラウザでの目視確認、外部サービス連携等）は `- [ ]` のまま残す
   - 更新したチェックリストで再度 `gh pr edit` を実行し、GitHub上のPR bodyを更新する

## 注意事項

- Git CLIを使用してPR情報を取得する
- テンプレートが存在する場合は必ず参照する
- 文体はフォーマルで、敬語は使わない
- **差分を一つ一つ丁寧に確認し、変更内容を正確に把握すること**
- 基本情報（PR番号、作成者、作成日、URL等）は含めない
- 単なるファイル名の羅列ではなく、変更の内容・意図を具体的に記述
- 既存のアーキテクチャやコーディングパターンを理解した上で記述
