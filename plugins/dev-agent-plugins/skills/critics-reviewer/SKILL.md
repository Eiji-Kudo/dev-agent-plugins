name: critics-reviewer
description: サブエージェントを使ってPRを専門レビュワーで並列に批判的レビューする
---

実行環境のサブエージェント機能を使い、PRの変更内容を専門レビュワーで並列に批判的レビューする。
エージェントの数もtokenも惜しまず使うこと。コンテキスト収集・レビュー・相互検証・解説充実の各フェーズで、必要と思われるだけ並列エージェントをふんだんに起動する。
レビュワーの人数・専門領域はPRの内容に応じて動的に決定する。
レビュワーの結果をリードが統合し、相互検証の判断を行う（competing hypotheses パターン）。

## 引数

`$ARGUMENTS`

- **指定あり**: その値をPR番号として使用する
- **未指定**: `gh pr view` で現在のブランチに紐づくPRを自動取得する

## 実行フロー

### 1. PR情報の取得

```bash
gh pr view [PR番号] --json number,title,body,url,headRefName,baseRefName
gh pr diff [PR番号]
```

PRが見つからない場合はその旨を伝え、処理を終了する。
変更されたファイルの一覧も取得しておく。
diffはプロジェクトルートの `pr-<PR番号>-diff.txt` に保存し、各レビュワーが直接読めるようにする。

### 1.5. 既存レビュードキュメントの確認

以下の **2段階** で既存のcriticsレビュードキュメントを探す:

#### 段階1: ファイルシステム上の確認

以下のパスにファイルが存在するか確認する:

- `documents/critics-review-pr-<PR番号>.md`
- `documents/critics-review-pr-<PR番号>-backend.md`
- `documents/critics-review-pr-<PR番号>-frontend.md`
- プロジェクトルート直下の `critics-review-pr-<PR番号>.md`

#### 段階2: git履歴の確認（段階1で見つからなかった場合）

ファイルが現在のツリーに存在しない場合、過去のコミットで作成・削除されている可能性がある。以下のコマンドで確認する:

```bash
git log --oneline -- 'documents/critics-review-pr-<PR番号>.md' '*critics-review-pr-<PR番号>.md'
```

コミット履歴がある場合、最新の削除コミットの直前の内容を復元する:

```bash
git show <削除コミット>^:documents/critics-review-pr-<PR番号>.md
```

#### 既存ドキュメントが見つかった場合（段階1 または 段階2）:

- その内容を全文読み込み、前回の指摘事項・議論内容を把握する
- **既存ドキュメントを編集のベースとして使うこと**。空のテンプレートから再生成してはいけない
- **保存先は既存ファイルのパスを最優先すること**。同じ PR 番号の既存ファイルがあるのに、別パスへ新規作成してはいけない
- **レビュワー構成は現在のdiffから改めて判断すること**。前回と同じ構成が適切ならそれでよいが、思考停止で前回の構成をコピーしてはいけない。前回の修正により変更の重心が移っている可能性がある（例: セキュリティ面の指摘が修正済みで、残りはデータ整合性の懸念だけなら、レビュワー構成を見直すべき）。ステップ3の判断基準に従い、現在のdiffの実態に基づいて判断する
- 前回の指摘が現在のdiffで修正済みかどうかを確認する観点を、各レビュワーへの指示に含める
- 統合レポートでは、前回からの変化（解決済み・未解決・新規）を明示する
- **レビュー回数をインクリメント**する（例: 前回が6回目なら今回は7回目）
- 過去の対応済み・対応不要の懸念点はそのまま引き継ぎ、新規の懸念点のみ追加する
- 既存の未対応懸念点についても、本文・コード引用・議論メモを可能な限り保持し、必要箇所だけ更新する

#### 既存ドキュメントが見つからなかった場合（両段階とも該当なし）:

- 通常通り新規レビューとして進める

### 2. ガイドラインの収集

プロジェクトルートおよび変更ファイルが属するディレクトリから、以下のファイルを探索する:

- `CLAUDE.md` / `CLAUDE.local.md`
- `AGENTS.md`
- `CONTRIBUTING.md`
- `GUIDELINES.md`
- `README.md`
- 変更ファイルの親ディレクトリにある上記ファイル（例: `backend/AGENTS.md`）

見つかったガイドラインの内容を、後続のレビュワーへの指示に含める。

### 3. diff分析とレビュワー構成の決定

リードはPRのdiffと変更ファイル一覧を分析し、**どの専門領域のレビュワーが必要か**を判断する。

#### 判断基準

以下の観点でdiffを分析し、関連性の高い専門領域をふんだんに選定する:

| 変更の性質 | 関連する専門領域の例 |
|------------|----------------------|
| API endpoint、入力処理、認証・認可 | セキュリティ |
| DBクエリ、スキーマ変更、マイグレーション | データ整合性、パフォーマンス |
| ビジネスロジック、状態管理、エラー処理 | 正確性、堅牢性 |
| ファイル構成、モジュール分割、依存関係 | アーキテクチャ、設計 |
| UI/フロントエンド、コンポーネント | UX、アクセシビリティ |
| インフラ、CI/CD、設定ファイル | 運用、信頼性 |
| テストコード | テスト戦略、カバレッジ |

上記はあくまで参考例であり、リードはdiffの実態に基づいて自由にレビュワーの専門領域と人数を決定してよい。
ただし以下の制約を守ること:

- レビュワーの人数に上限を設けず、必要な専門領域の数だけ**ふんだんに**レビュワーを生成すること。少なくする理由がない限り多めに出すこと
- 各レビュワーの専門領域が重複しないこと
- レビュワー名は `{領域}-reviewer` の形式とすること（例: `security-reviewer`, `data-integrity-reviewer`）
- なぜその構成にしたかの理由を、統合レポートの「レビュワー構成」セクションに記載すること

### 4. コンテキスト収集（並列エージェント）

レビュワーの精度を高めるため、レビュー実行前に**コンテキスト収集エージェントを並列起動**する。

#### 起動方式

- Claude CodeではAgent、Codex CLIではcollaboration sub-agentなど、実行環境で利用可能なサブエージェント機能を使用
- 変更ファイルの領域ごと（例: API層、DB層、フロントエンド、テスト等）に1エージェントずつ割り当て、必要な数だけふんだんに並列起動する
- すべてのエージェントを1つのメッセージ内で同時に並列起動する

#### 各エージェントへの指示

- 担当する変更ファイルの周辺コードを調査し、以下を報告する:
  - 変更箇所が依存している関数・型・定数の定義と振る舞い
  - 変更箇所を呼び出している箇所（呼び出し元）
  - 関連するテストファイルの有無と内容
  - 同じパターンの既存実装（整合性チェック用）
- 報告は簡潔にまとめ、レビュワーへの参考情報として使える形式にする

#### リードの処理

各エージェントの報告を受け取り、コンテキスト情報としてレビュワーへの指示に組み込む。

### 5. 並列サブエージェントによるレビュー実行

決定したレビュワー構成に基づき、実行環境で利用可能な高性能モデルの**サブエージェントを並列に起動する**。
利用可能な並列呼び出しをまとめて行い、すべてのレビュワーを同時に起動すること。

各レビュワーの Agent プロンプトには以下を含めること:
- レビュワーの役割と専門領域
- PRの概要（title, body）
- diffファイルのパス（プロジェクトルートの `pr-<PR番号>-diff.txt`）を指定し、レビュワーが自分で読むよう指示
- 収集したガイドラインの要約
- そのレビュワーが担当する具体的なチェック項目
- ステップ4で収集したコンテキスト情報（担当領域に関連する部分）
- **既存レビューがある場合**: 前回の対応済み/対応不要の懸念点リストを含め、再指摘しないよう指示

#### 各レビュワーへの共通指示

- ultrathinkを使って深く思考すること
- diffファイルを自分で読み、必要に応じてソースファイルも直接読むこと
- 懸念点のみを抽出し、問題のない箇所にはコメントしない
- 各懸念点に重要度（CRITICAL / HIGH / MEDIUM）を付与すること
- 該当ファイル名と行番号を特定すること
- **問題点の説明は詳しく書くこと**: 何が問題か、どういうシナリオで顕在化するか、該当コードの引用を含めて具体的に説明する。「〜の可能性がある」だけでなく、再現手順や条件を明記する
- **推奨対応には具体的な修正例を含めること**: 方向性だけでなく、修正後のコードスニペットを提示する。読み手がそのまま実装に移れるレベルの具体性を持たせる
- 確信度が低い懸念点は明示的にその旨を記載すること
- コードの変更は一切行わない。懸念点の報告のみ行う
- **スコープの制約**: 懸念点はPRの変更によって生じた、または顕在化した問題に限定する。PRの変更と因果関係がない既存の問題は報告しない。diffに含まれないファイルについて指摘する場合は、PRの変更との因果関係を明示すること
- **既存レビューがある場合**: 前回の指摘が修正済みか未対応かを確認し、新規の懸念点と区別して報告すること

### 6. 相互検証（並列エージェント）

全レビュワーの結果が揃った後、**相互検証エージェントを並列起動**し、各懸念点の妥当性を他の専門領域の観点から検証する。

#### 起動方式

- 実行環境で利用可能なサブエージェント機能を使用
- 検証の観点ごとにエージェントをふんだんに起動する（例: 「矛盾検出」「重複検出」「確信度低の検証」「コード実態との照合」等）
- すべてのエージェントを1つのメッセージ内で同時に並列起動する

#### 各エージェントへの指示

- 全レビュワーの発見一覧を渡す
- 担当する検証観点を明示する:
  - **矛盾検出**: レビュワー間で矛盾する指摘がないか確認し、どちらが正しいか判定する
  - **重複検出**: 異なるレビュワーが同一の問題を別の角度から指摘していないか確認し、マージ案を提示する
  - **確信度検証**: 確信度が低い懸念点について、該当コードを直接確認し、採否を判定する
  - **コード照合**: 懸念点で引用されているコードが実態と一致しているか、行番号が正しいか確認する
- 必要に応じてソースファイルを直接読み、判断の根拠を示すこと

#### リードの統合

各エージェントの検証結果を統合し、最終的な懸念点一覧を確定する。

### 7. 結果の統合

相互検証の結果を踏まえ、リードが最終レポートを作成する:
- 重複する懸念点はマージ
- 矛盾がある場合はリードの判断と理由を記載
- 確信度が低い懸念点は採否の理由を記載
- 重要度順に並べ替え

### 8. 解説充実フェーズ（並列サブエージェント）

統合レポートの懸念点一覧が確定した後、**解説充実用のサブエージェントを並列起動**し、各懸念点の「問題点」と「推奨対応」の記述を具体化する。

#### 起動方式

- 実行環境で利用可能なサブエージェント機能を使用
- 懸念点を均等に分割し、ふんだんにサブエージェントを並列起動する。エージェント数を節約しようとせず、惜しまず使うこと
- すべてのサブエージェントを1つのメッセージ内で同時に並列起動する

#### 各サブエージェントへの指示

- 担当する懸念点の一覧（統合レポートから該当部分を抜粋）
- diffファイルのパス（`pr-<PR番号>-diff.txt`）
- 以下の2点を充実させること:

**問題点の充実**:
- 該当コードを引用し、何が問題かをコード上で示す
- どういう操作・条件・タイミングで問題が顕在化するかの具体的シナリオを記述する
- 影響範囲（他のコンポーネントやフローへの波及）があれば記述する

**推奨対応の充実**:
- 修正後のコードスニペットを提示する（そのまま適用できるレベル）
- 修正によって何が改善されるかを簡潔に説明する
- 複数の選択肢がある場合は比較して推奨を示す

#### サブエージェントへの制約

- 懸念点の追加・削除・重要度の変更はしない
- 既存の構造（番号、タイトル、検出者等）は変更しない
- コードの変更は一切行わない
- 出力は担当懸念点ごとに「問題点」と「推奨対応」の充実版テキストを返す

#### リードの統合

サブエージェントの結果を受け取り、統合レポートの各懸念点に反映してドキュメントを完成させる。

## リードの行動規則

- 自分ではコードレビューを行わない（レビュワーに委任する）
- 全レビュワーの完了を待ってから統合に入る
- 統合時は重複排除と矛盾解決に集中する
- コードの変更は一切行わない。すべての懸念点を報告し、修正はユーザーに委ねる

## 重要度の定義

| 重要度 | 意味 |
|--------|------|
| CRITICAL | 必ず修正すべき深刻な問題（セキュリティ、データ破損） |
| HIGH | 修正を強く推奨する問題（バグ、重大なパフォーマンス問題） |
| MEDIUM | 検討を推奨する問題（設計改善、軽微なパフォーマンス問題） |

## ファイル保存

### 保存先の決定

**情報保持を最優先すること。**

- 同じ PR 番号の既存ファイルがある場合は、そのファイルを更新し、保存先を変えない
- 既存が単一ファイルなら、今回の懸念点が増えても**勝手に分割しない**
- 既存が分割ファイルなら、その分割構成を維持する
- 構成変更（単一→分割、分割→単一）は、既存ファイルが存在しない新規作成時だけ許可する
- 「documents/ があるから移動する」のようなパス変更は禁止。蓄積済み情報を優先する

PRの変更ファイルがフロントエンド・バックエンドの両方に跨り、懸念点が多い場合（目安: 合計10件以上）、ドキュメントを分割する:

- `documents/critics-review-pr-<PR番号>-backend.md` — バックエンド関連の懸念点
- `documents/critics-review-pr-<PR番号>-frontend.md` — フロントエンド関連の懸念点

分割の判断基準:
- 変更ファイルが `backend/` と `frontend/` の両方に存在する
- 懸念点の総数が多く、1ファイルでは見通しが悪い
- 各懸念点は、該当ファイルのパスに基づいてバックエンド/フロントエンドに振り分ける
- 両方に関わる横断的な懸念点は、より関連の深い方に配置する

分割時の採番規則:
- **フロントエンド**: 1番から連番
- **バックエンド**: 101番から連番
- これにより、ドキュメント間で番号が重複しない

分割しない場合（片方のみの変更、または懸念点が少ない場合）:
- `documents/critics-review-pr-<PR番号>.md` に単一ファイルとして保存する

`documents/` ディレクトリが存在しない場合はプロジェクトルートに保存する。

### 既存ドキュメントがある場合の保存方式

ドキュメントを**上書き更新**する。構造は以下のようにする:

1. **ヘッダー部分**（概要・レビュワー構成・参照ガイドライン）は最新の情報で更新する
2. **未解決の懸念点**をドキュメント上部の「未対応の懸念点」に `<details>` トグルで配置する
3. **対応不要の懸念点**（全員合意で「このPRでは対応不要」と判断されたもの）は折りたたみセクションに移動する
4. **対応済みの懸念点**（実際にコード修正された or 確認済みのもの）は折りたたみセクションに移動する
5. **既存の懸念点本文は消さない**。問題点・該当コード・推奨対応・チーム内議論は、更新が必要な箇所だけ差し替え、それ以外は保持する
6. **新規懸念点は追記する**。既存の懸念点番号を振り直したり、既存項目を丸ごと作り直したりしない

絶対ルール:

- 既存ファイルがあるのに、内容を全削除して新規テンプレートで作り直してはいけない
- 既存の `<details>` ブロックを、理由なく丸ごと消してはいけない
- 既存の「対応済み」「対応不要」セクションは、内容を要約し直さず原文を保持する
- 重複統合が必要でも、古い記述を完全に捨てず、どちらを採用したか分かる形で反映する

具体的な構成:

```markdown
## サマリー

| 重要度 | 件数 | 対応済み |
|--------|------|----------|
| CRITICAL | N | N |
| HIGH | N | N |
| MEDIUM | N | N |

## 未対応の懸念点

<details>
<summary>1. [懸念点のタイトル]（HIGH / 未対応）</summary>

（懸念点フォーマット）

</details>

---

## 対応不要の懸念点

<details>
<summary>3. yyy（MEDIUM / 対応不要）</summary>

（元の懸念点内容をそのまま保持）

</details>

## 対応済みの懸念点

<details>
<summary>2. xxx（CRITICAL / 修正済み）</summary>

（元の懸念点内容をそのまま保持）

</details>
```

- **すべての懸念点を `<details>` トグル形式にする**（未対応を含む）
- 各トグルの `<summary>` は `{番号}. {タイトル}（{重要度} / {対応状況}）` の形式
- 番号は元の番号を維持する（振り直さない）
- 再レビューのたびに、修正が確認できた項目・対応不要と合意された項目を下部の該当セクションに移す
- 「対応不要」と「対応済み（修正済み）」は別セクションに分ける
- 本文の更新は **in-place 編集** を基本とし、既存情報の保持を優先する
- 新しい懸念点の番号は、既存番号を避けて末尾に追加する
- 既存のレビュー履歴や議論が価値を持つ場合は、`チーム内議論` や注記として残す

### 新規の場合

- 通常の出力フォーマットで新規作成する（対応不要・対応済みセクションは空で生成しない）
- すべての懸念点を `<details>` トグル形式で出力する

保存後、パスを簡潔に報告する。

### クリーンアップ

レビュードキュメントの保存が完了したら、プロジェクトルートの `pr-<PR番号>-diff.txt` を削除する。

## 制約事項

- **懸念点のみ抽出**: 良い点や問題のない箇所についてはコメントしない
- **コード修正は一切行わない**: レビュワー・リードともにコードを変更しない。軽微なものも含め、すべての懸念点は報告のみ行い、修正判断はユーザーに委ねる
- **フォローアップ issue はクリック可能なリンクで併記**: 懸念点を follow-up issue に切り出した（PR スコープ外として別 issue 化した）場合は、その懸念点（多くは「対応不要」セクション）に **クリックで遷移できる Markdown リンク** `[#<番号>](https://github.com/<owner>/<repo>/issues/<番号>)` を必ず併記する。`#<番号>` だけのプレーンテキストにしない
- **日本語で記述**: すべての出力は日本語で記述する
