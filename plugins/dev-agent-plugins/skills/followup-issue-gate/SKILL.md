# follow-up issue 作成ゲート（オーバーエンジニアリング防止）

作業フロー（PR 対応・レビュー分析・調査など）の途中で follow-up issue（別 issue への切り出し）を作る前に、必ず以下を確認する。条件を満たさないものは issue 化しない。

## 作成条件（すべて満たすこと）

- **具体的な実害がある**: bug / regressions / security / data loss / type safety / maintainability drift / operational risk のいずれかとして、放置した場合の悪影響をコード・ログ・CI・仕様差分で説明できること。
- **根拠が現在の repo 状態で検証済み**: 推測・好み・将来の拡張余地だけでは作らない。該当ファイル/行、再現手順、失敗するテスト、既存仕様との差分のいずれかを issue 本文に書けること。
- **今PRで直さない理由が明確**: 今PRの目的から外れる、影響範囲が大きい、別リリース順序が必要など、分離する理由を説明できること。単に「ついでに後でやると良さそう」だけでは作らない。
- **解決策が過剰設計ではない**: issue 本文は「問題」と「最小の受け入れ条件」を中心に書き、抽象基盤・共通化・設定化・プラグイン化・大規模リライトを前提にしない。必要ならまず局所修正・既存パターン・生成型/既存SoTの利用を第一候補にする。
- **1 issue で完結する粒度**: 複数の独立した改善を束ねない。逆に、実害の薄い細切れ改善を量産しない。

## issue 本文のルール

follow-up issue を作る場合、本文はエンジニアだけでなく biz / product / ops の人にも価値が伝わるように書く。最低限、以下を含める:

- **なぜ今やる価値があるか**: ユーザー体験、運用負荷、売上機会、信頼性、レビュー速度、将来の開発速度など、事業・プロダクト上の利益を1〜3文で書く。
- **放置した場合の困りごと**: 技術用語だけでなく、「通知が届かず機会損失につながる」「調査に毎回時間がかかる」「誤った内容を配信するリスクが残る」のように、非エンジニアが判断できる影響に言い換える。
- **技術的な根拠**: 該当ファイル/行、再現手順、失敗するテスト、既存仕様との差分など、判断材料を具体的に書く。
- **受け入れ条件**: 完了時に何ができていればよいかを、確認可能な箇条書きで書く。実装手段を固定しすぎず、期待する状態を中心にする。
- **今PRに含めない理由**: 目的がぼやける、影響範囲が大きい、リリース順序を分けたいなど、分離理由を明記する。

詳細な調査ログ・代替案・コード上の技術メモは削除しなくてよい。ただし、issue 上部に「なぜやるか」「何が終わればよいか」が分かる要約を置き、技術メモは `## 技術メモ` / `## 補足` / `## 調査ログ` として下部に分離する。

issue タイトルも、内部都合だけでなく利益が伝わる名前にする。例: `通知失敗時に原因を切り分けやすくして運用対応時間を減らす`、`公開前のデータ形式崩れを検知して配信品質を守る`。

## issue 化しないもの

- 「将来使うかもしれない」抽象化、設定化、共通化
- 現在のコードで発生条件を説明できない仮説上の edge case
- 既存パターンに反していない好みのリファクタリング
- CI / lint / type check / test が既に担保していて、人間が追跡する価値が薄いもの
- PR の目的と無関係な探索中に見つけた軽微な改善案（必要なら最終報告のメモに留める）

## 例外（プロジェクトガイドラインで必須のもの）

プロジェクトの CLAUDE.md / AGENTS.md 等で issue 化が必須とされているもの（例: 型安全性を損なう箇所の記録）は作成してよい。ただし、その場合も上記ゲートのうち「根拠が現在の repo 状態で検証済み」「解決策が過剰設計ではない」「1 issue で完結する粒度」を満たし、問題の事実と最小の修正方針だけを書く。
