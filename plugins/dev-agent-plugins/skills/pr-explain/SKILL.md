name: pr-explain
description: PR全体の解説md作成（mermaid図含む）
---
PRの内容を解説するMarkdownを作成してください。

引数:
- `$ARGUMENTS` にPR番号があればそのPRを対象にする（例: `553`, `#553`, `pr-553`, GitHub URL いずれも許容）
- 引数が無ければ `gh pr view --json number` で現在ブランチのPRを使う
- 現在ブランチにPRが無く、引数も無ければエラー終了

## 手順

1. **PR番号の解決**
   - 引数があれば数値部分を抽出。URL の場合は末尾の数字を取る
   - 無ければ `gh pr view --json number -q .number`
   - 見つからなければ「対象PRが特定できない」と報告して終了

2. **PRメタデータと差分の取得**
   - `gh pr view {番号} --json title,body,baseRefName,headRefName,additions,deletions,changedFiles,files,state,url,author`
   - `gh pr diff {番号}` で差分を取得（リネーム検出が必要なら `gh pr checkout` 後に `git diff -M50 {base}...HEAD`）
   - 差分が大きい場合 (>1000行 目安) は Read で chunk 読み or `gh pr diff --name-only` でファイル一覧を取り、 重要ファイルだけ深掘りする

3. **PR本文・関連リンクの整理**
   - PR description の Summary / 親 issue / 関連PR / closes 表記をメモ
   - 必要なら `gh issue view {番号}` で参照 issue の中身も拾う
   - 「土台のみ / 後続PRで実装」など段階的変更の前提が書かれていたら必ず明記する

4. **アーキテクチャ把握**
   - 変更対象ディレクトリの周辺コード・CLAUDE.md を Read で確認し、 既存パターンとの差を理解する
   - 新規ファイルは全文を、 既存ファイルは差分を中心に読む

5. **解説mdの構成**

   以下のセクションを基本骨子とし、 PRの性質に合わないセクションは削る（無理に埋めない）。

   ```
   # PR #{番号} 解説: {タイトル}

   - PR: {url}
   - branch: {head} → {base}
   - 規模: +{additions} / -{deletions}, {changedFiles} files
   - 親issue / 関連PR / closes
   ```

   1. **一行で言うと** — レビュアーが3秒で趣旨を掴める1段落
   2. **全体像 (mermaid)** — 環境/サービス/データフローの俯瞰図
   3. **ファイル別 変更マップ (mermaid)** — どのファイルに何が入ったか
   4. **主要ファイルごとの解説** — 新規ファイルは構造、 既存ファイルは差分の意図
   5. **追加リソース/エンティティ表** — Terraform plan・migration・新APIなど数えられるものは表で
   6. **ER 図 (mermaid erDiagram)** — **DB のテーブル/スキーマ/migration を変更する PR では必須**。変更したテーブルに加え、FK で関連するテーブル（参照先・参照元）も含めて relationship を描く。追加/削除/型変更したカラム・制約など「この変更で何が変わったか」が ER 図と本文から分かるようにする（変更テーブルを中心に 1〜2 hop に絞ってよい）
   7. **依存・適用フロー (mermaid)** — apply / deploy / migration / feature flag の段取り
   8. **既存パターンとの差分（意図的に残しているもの）** — コピー元との diff があれば表で
   9. **レビュー観点** — チェックリスト形式 (`- [ ]`)。 自明でないもの・運用判断が要るものに絞る
   10. **フォローアップ issue** — この PR の作業中に作成した follow-up issue（critics の対応不要→別 issue、型安全性/ドリフトの issue 化、レビュー由来の別 issue 等）を **クリックで遷移できる Markdown リンク** `[#<番号> <タイトル>](https://github.com/<owner>/<repo>/issues/<番号>)` で列挙する。`#<番号>` だけのプレーンテキストにせず必ず URL リンクにする。作成した issue が無ければ「フォローアップ issue: なし」と明記し、セクションを省略しない

6. **mermaid 利用ルール（重要）**
   - **色は使わない**（`fill:` / `stroke:` / `classDef` での着色を入れない）
   - 図は1枚あたり12〜15ノードまでを目安に。 巨大な1枚図より、 観点別に分けた小さな図を複数枚にする
   - **ラベルは記号を含むなら必ずダブルクオートで囲む**: ノードラベル・subgraph タイトルに `<br/>` や記号（`+` `-` `/` `:` `(` `)` 等）があると、括らないと `+` `/` `-` でパースエラー（確認済み）。`["a<br/>b"]` / `subgraph id["title +1 -2"]`。迷ったら囲む
   - **矢印ラベルはパイプ形式で書く**: `A -->|text| B` / `A -.->|text| B`。インライン太線 `A == text ==> B` は **mermaid 9 で parse error**（閲覧側が古い版のとき壊れる。確認済み）。太線が必須でなければ `-->` で十分
   - **凝った形状を避ける**: 円柱 `[(...)]` などは描画されないことがある。原則は矩形 `[...]`。描画崩れの切り分け時はまず矩形に置換する
   - シーケンス図 / フローチャート / ER図 / state図のうち、 内容に合うものを選ぶ。 とりあえず `graph LR` で済ませない
   - **DB のテーブル/スキーマ/migration を変更する PR では `erDiagram` を必ず 1 枚入れる**: 変更したテーブル ＋ FK で関連するテーブル（参照先・参照元）を描き、PK/FK と主要カラムを示す。リレーションは `||--o{` 等で多重度を表す。エンティティ名・属性に記号があればクオートする。関連が多い場合は変更テーブル中心に 1〜2 hop へ絞る。「変更テーブルが孤立した 1 個だけの ER 図」にせず、関連テーブルとの線まで描くこと
   - 矢印は `-->`（実線=直接依存）と `-.->`（点線=参照/data source/将来予定）を意味分けして使う

7. **コードブロック整形ルール**
   - HCL / YAML / TS など複数行が前提の言語は **1行に詰めて要約しない**。 元コードの改行・インデントをそのまま維持する
   - スペースで揃えた多重スペース (`name     = "x"` を 1 行で連ねる等) で「圧縮」しない。 縦に並べて読めることを最優先
   - 長すぎる場合は省略 (`# ...`) で省く方が、 詰めて貼るより読みやすい

8. **保存先**
   - **`.local` には置かない**（user CLAUDE.md ルール）
   - プロジェクトに `documents/` があれば `documents/` 配下の **テーマに合うサブディレクトリ** に置く（例: terraform PRなら `documents/terraform/`, frontend PRなら `documents/frontend/`）
   - 対応するサブディレクトリが無く適切な置き場が判断できなければプロジェクトルートに置く
   - ファイル名は `pr-{番号}-{slug}.md`（slugはタイトルから3〜5語の英小文字 kebab-case を抽出）
   - 既に同名ファイルがある場合は上書き前に diff を取り、 ユーザに上書き確認

9. **mermaid レンダリング検証（必須・報告前に実施）**
   - md を書いたら、各 ` ```mermaid ` ブロックを取り出して `mmdc` でレンダリング検証する。**全図が成功するまでユーザに報告しない**
   - 例: ブロックを `/tmp/d.mmd` に書き出して `mmdc -i /tmp/d.mmd -o /tmp/d.svg`。`command not found` のときは nodenv の対象 Node 版を指定（エラー出力に表示される）: `NODENV_VERSION=<ver> mmdc -i ...`
   - **重要**: `mmdc` は mermaid 11 を使うが、閲覧側（VS Code preview 等）は mermaid 9/10 のことがある。CLI で OK でも古い版で壊れうるので、上記「ラベルはクオート / 矢印はパイプ形式 / 凝った形状を避ける」を必ず守る。可能なら mermaid 9 系の `parse` でも確認する
   - 1 図でも壊れていると preview 全体が描画されないことがあるため、**全図**を検証する

8. **作成後の報告**
   - 保存先パスを1行で報告
   - 何を mermaid 図にしたか箇条書きで列挙（ユーザがすぐ図の妥当性を確認できるように）
   - PRの「未解決 / 後続PR待ち」項目があれば最後に明記

## 注意事項

- 解説対象はPR本人。 自分の意見・推測でPRに無い前提を足さない
- description の主張と差分の実態が食い違ったら、 **差分を正** とし、 食い違いを「レビュー観点」に挙げる
- 公式ドキュメント・料金・SLA など外部知識に依存する記述は、 不確かなら断定しない（user feedback memory: verify-before-answering）
- 個別ファイルの羅列より「機能単位 / 目的単位」でグルーピングして説明する
- コミットには触れない（コミット解説ではなくPR解説。 historyは `git log` で見れば足りる）
- 解説md内で対象PRの description を丸コピしない。 description は読まれている前提で、 **構造化と図** で価値を足す
