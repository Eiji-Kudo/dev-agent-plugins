critics reviewファイル（`critics-review-pr-*.md` および `*-resolved.md`）を、**ユーザーが確認すべき判断ポイント一覧**に書き換える。loop-critics-fix の事後ログではない。

## このファイルが伝えたいこと

resolved ファイルの読み手 = 後から PR を見直すユーザー。読み手が知りたいのは:

1. **Claude が独断で下した判断のうち、自分が承認 or 差し戻しすべきもの**
2. （副次的に）どういう思想でどこを直したか（深掘り用）

書いてはいけないもの:

- loop-critics-fix が何イテレーション回ったか等のメタ情報
- 自明な修正の個別説明（タグ追加、description 修正、誤字訂正等）
- レビュワーがどう指摘したかの再掲

## 引数

`$ARGUMENTS`

- **指定あり**: そのパスのファイルを対象とする
- **未指定**: 現在のブランチのPR番号から対象ファイルを特定する
  1. `gh pr view --json number` で現在のブランチに紐づくPR番号を取得する
  2. `**/critics-review-pr-{PR番号}*.md`（resolved 含む）で検索する
  3. 候補が複数 → ユーザーに選択を求める（resolved とそうでないものを両方提示）
  4. 候補が0件 → ユーザーにパスを指定するよう求める

## 出力フォーマット

```markdown
# PR #{番号} 確認待ちの判断

> 未対応の懸念点は [critics-review-pr-{番号}.md](./critics-review-pr-{番号}.md) を参照（resolved ファイルの場合のみ）

## 確認してほしいこと

### {通し番号}. {判断のタイトル}

- **Claude の判断**: {何をどう判断したか / どう修正 or 対応不要にしたか}
- **確認してほしい点**: {ユーザーがOKを出すべきポイント。「この方針で良いか」「対応不要判断は妥当か」「別案A/Bのうち選んだのはこれで良いか」など具体的に}

（確認待ちが0件なら「確認待ち項目なし」と明記する）

<details>
<summary>修正の思想と内容（付録）</summary>

### 思想

- {修正全体を貫く方針・スタンスを2-4行で。例: 「private repo 前提でセキュリティ系は実害ベースで判断」「upstream 仕様に合わせて X を Y に統一」}

### 修正サマリ

| # | 重要度 | 内容 | 修正 |
|---|--------|------|------|
| {番号} | {CRITICAL/HIGH/...} | {1行要約} | {修正内容1行} |
| {複数の軽微な修正の番号} | LOW | {まとめた説明} | 修正済み |
| {対応不要の番号} | - | 対応不要 | {理由を1行} |

</details>
```

通常の critics ファイル（未対応含む）の場合:

- ヘッダーは `# PR #{番号} 懸念点サマリ`、未対応リンクは付けない
- 「確認してほしいこと」は対応済み・対応不要のもののみ対象（未対応は loop-critics-fix の責務）
- テーブルの `修正` 列を `状態` 列にし、`未対応 / 対応中 / 対応不要 / 修正済み` を記載する

## 「確認してほしいこと」に載せる判断基準

以下のいずれかに該当するものは載せる:

- **「対応不要」と Claude が独断で判断したもの**（YAGNI、private repo前提、dev環境のみ等の理由付け）
- **複数の修正方針があったうち1つを選んだもの**
- **推奨対応をそのまま採用せず、別の形で修正したもの**
- **設計判断・スコープ判断を伴うもの**（既存コードのリファクタ範囲、命名、抽象化レベル等）
- **silent failure / upstream 仕様誤解 / 環境間不整合** など、判断の前提が間違っていた可能性があるもの

該当しない（タグ追加・description修正・誤字訂正・自明な型エラー修正等）は付録テーブルにのみ載せる。

## 実行フロー

1. 対象ファイルを読み込む
2. 各懸念点を分類する: 「確認してほしい判断」 or 「自明な修正（テーブル送り）」
3. 上記フォーマットで書き換える
4. **並列サブエージェントで読みやすさを self refine する**（後述）
5. 報告する: 確認待ちN件、付録テーブルM件、行数の削減率、refine で直した点

### 並列サブエージェントによる self refine

書き換え後のファイルを、実行環境で利用可能なサブエージェント **2体に並列で**ユーザー視点から読み返してもらう。各エージェントには以下を伝える:

- 役割: 「PR を後から見直すユーザー」として、書き換え後の resolved ファイルを読む
- 渡すもの: 書き換え後のファイルパス、元ファイルの主要懸念点リスト（番号と1行要約）
- 評価軸:
  1. **「確認してほしいこと」がそのままユーザーの判断アクションに繋がるか**（曖昧で「結局何をOKすればいいか分からない」項目はNG）
  2. **付録テーブルだけで済ませた項目に、本当はユーザー確認が要るものが紛れていないか**
  3. **思想セクションが「修正全体を貫く方針」になっているか**（個別修正の羅列になっていないか）
  4. **冗長な再掲・メタ情報が残っていないか**
- 出力: 修正提案を箇条書きで（「N番の確認ポイントは〜と書き直すべき」「N番は付録ではなく確認待ちに昇格すべき」等）

2体の指摘を統合し、ファイルを再編集する。指摘が衝突する場合は「ユーザーが確認アクションを取れるか」を優先軸に判断する。指摘が0件なら refine スキップ。

## 制約事項

- **元ファイルを上書きする**（別ファイルは作らない）
- **番号は元のドキュメントから維持する**（振り直さない）
- **resolved ファイルの場合、未対応の懸念点への参照リンクは維持する**
- **付録は必ず `<details>` で畳む**（メイン視点を埋もれさせない）
- **日本語で出力する**
