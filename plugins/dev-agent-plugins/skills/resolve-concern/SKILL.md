---
name: resolve-concern
description: レビュードキュメントの懸念点を対応済みに更新する
---

コード修正後に、対応するレビュードキュメント（critics review / review-decisions）の懸念点を「未対応」から「対応済み」に移動し、サマリーカウントを更新する。

## 引数

`$ARGUMENTS`

- **指定あり**: 対応済みにしたい懸念点を特定する文章（懸念点のタイトルやキーワード）
- **未指定**: 直近の `git diff --cached` および `git diff` から変更内容を分析し、対応する懸念点を推定する。候補が複数ある場合はユーザーに選択を求める。

## 実行フロー

### 1. レビュードキュメントの探索

`gh pr view --json number` で現在のブランチに紐づくPR番号を取得し、以下のパターンで検索する:

- `**/critics-review-pr-{PR番号}*.md`
- `**/review-decisions-pr-{PR番号}*.md`
- `**/test-design-pr-{PR番号}*.md`

候補が複数見つかった場合は、すべてのファイルを対象とする。
PR番号が取得できない場合は、ユーザーにパスを指定するよう求める。

### 2. 対象懸念点の特定

見つかったドキュメントの未対応項目から、引数またはdiff内容に一致する項目を特定する。

未対応項目の探索場所はドキュメント構造により異なる:

- **critics-review**: 「未対応の懸念点」セクションから探す
- **review-decisions（トリアージ済み）**: 「未対応の懸念点」セクションから探す
- **review-decisions（トリアージ前）**: MUST_FIX / SHOULD_CONSIDER セクションから、`<summary>` に「対応済み」「修正済み」「対応不要」を含まない `<details>` ブロックを探す（CAN_IGNORE セクションは対象外）
- **test-design の場合**: 「未実装のテスト項目」セクションからテスト項目を探す

#### 特定方法

- **引数指定時**: タイトル・キーワード・ファイル名の一致で特定
- **引数未指定時**: `git diff HEAD` の変更ファイルと各項目の対象ファイルを照合し、候補を絞り込む。テスト設計の場合は、追加されたテストファイルと配置先の一致も照合する。

#### 候補が0件の場合

「未対応の項目に一致するものが見つかりませんでした」と報告して終了する。

#### 候補が複数ある場合

一覧を表示してユーザーに選択を求める。

### 3. 修正内容の要約生成

`git diff` および `git log` から、対象懸念点に関連する変更内容を分析し、簡潔な修正サマリーを生成する。

サマリーのフォーマット:
- 何を変更したか（1-2文）
- 具体的な修正内容（箇条書きが適切な場合は箇条書き）

### 4. ドキュメントの更新

#### 4a. 懸念点の移動

対象項目を元のセクションから削除し、「対応済みの懸念点」セクションに**要約形式**で移動する。

- **critics-review**: 「未対応の懸念点」セクションから削除 → 「対応済みの懸念点」セクションへ
- **review-decisions（トリアージ済み）**: 「未対応の懸念点」セクションから削除 → 「対応済みの懸念点」セクションへ
- **review-decisions（トリアージ前）**: MUST_FIX / SHOULD_CONSIDER セクションから `<details>` ブロックを削除 → 「対応済みの懸念点」セクションへ。セクション自体は残す（他の未対応項目がある場合があるため）。セクション内の全項目が対応済みになった場合でもセクション見出しは残す。
- 「対応済みの懸念点」セクションが存在しない場合は、ドキュメント末尾に新規作成する

##### critics-review / review-decisions の場合

- `<summary>` のステータスラベルを変更:
  - コード修正で解消した場合: `未対応` → `修正済み`
  - 対応不要と判断された場合: `未対応` → `対応不要`

```markdown
<details>
<summary>{番号}. {タイトル}（{重要度} / 修正済み）</summary>

- **ファイル**: `対象ファイルパス`
- **問題**: 問題の要約（1文）
- **対応**: 修正内容の要約（1-2文）

</details>
```

- 問題点・修正内容はそれぞれ1-2文で簡潔に要約する
- コードスニペットは含めない（必要なら修正内容の説明文で十分）
- メタデータテーブル（重要度・検出者等）は省略する
- **review-decisions ドキュメントの場合**: 「返信案」セクションがあれば、実際の修正内容を反映した返信文に更新する

##### test-design の場合

- ステータスラベルを変更: `未実装` → `実装済み`
- テスト項目を「未実装のテスト項目」セクションから「実装済みのテスト項目」セクションに移動する

```markdown
<details>
<summary>{テスト名}（{優先度} / 実装済み）</summary>

- **対象**: `対象ファイルパス`
- **テストファイル**: `実際に作成/更新したテストファイルパス`
- **実装内容**: 実装したテスト内容の要約（1-2文）

</details>
```

- 元のシナリオ詳細は要約に置き換える
- 実際に作成・更新したテストファイルのパスを記載する

#### 4b. サマリーテーブルの更新

ドキュメント上部のサマリーテーブルの「対応済み」列を +1 する。

例:
```
| MEDIUM | 5 | 1 |  →  | MEDIUM | 5 | 2 |
```

#### 4c. 対応済みセクションでの配置

元のドキュメントでの番号順を維持する。既存の対応済み懸念点の間に、番号順で正しい位置に挿入する。

### 5. resolvedファイルの要約

更新後、`../summarize-resolved/SKILL.md` を読み、その手順に従って resolved ファイルを要約する。

### 6. 結果の報告

更新完了後、以下を簡潔に報告する:

- 更新したドキュメントのパス
- 対応済みにした懸念点のタイトルと番号
- 修正サマリーの内容

## 制約事項

- **対応済みセクションは要約形式**: 元の詳細な記述（コードスニペット、推奨対応等）は要約に置き換える。未対応セクションからは完全に削除する。
- **番号は振り直さない**: 元のドキュメントの番号体系を維持する。
- **「修正済み」と「対応不要」の両方に対応**: ユーザーが対応不要と判断した場合はステータスを「対応不要」とし、理由を記載する。コード修正で解消した場合は「修正済み」とする。
- **test-design の場合**: ステータスは「実装済み」を使用する。サマリーテーブルの「実装済み」列を +1 する。
- **日本語で報告**
